# PayTracker MVP

Рабочий прототип трекера заявок на платежи/закупки/консультации: FastAPI + Jinja2 +
SQLAlchemy 2.0 + PostgreSQL + Alembic. Реализует весь жизненный цикл заявки из
`статусы заявки.md`, справочник документов, делегирование полномочий Руководителя,
курс ЦБ с fallback, аудит-лог.

## Как запустить

```bash
docker compose up --build
```

Открыть http://localhost:8000 — попадёшь на страницу логина со списком демо-пользователей
(упрощённая авторизация без пароля — просто выбор пользователя, см. "Ограничения" ниже).

При первом запуске `entrypoint.sh` сам:
1. дожидается готовности БД,
2. накатывает `alembic upgrade head`,
3. засеивает справочники (валюты, подразделения, агенты, компании-покупатели, типы
   документов) и 5 демо-пользователей (1 Руководитель, 2 Исполнителя, 2 Заказчика).

## Что реализовано

- Создание заявки всех трёх типов (платёж / закупка / консультация) с полями из
  `заявки на платежи.md` / `заявки консультации.md`.
- Полная статусная машина из `статусы заявки.md`: Черновик → Новая → В работе ⇄
  Уточнение → Ожидает подтверждения → Проверка документов → Закрыта, плюс ветка
  Отклонена → Архив. Права на каждый переход проверяются по роли.
- Делегирование полномочий Руководителя (`PayTracker.md`) — делегат получает права
  назначать/отклонять заявки, действие помечается в истории и аудите.
- Загрузка документов и проверка комплектности по `справочник типов документов.md`
  перед закрытием заявки (обязательные/условные документы).
- Курс ЦБ с кэшем и fallback на устаревший курс при недоступности cbr.ru
  (`заявки на платежи.md`, раздел Edge case).
- Комментарии к заявке.
- Неизменяемый аудит-лог всех переходов статусов (`аудит.md`).

## Что НЕ реализовано (осознанно, за рамками MVP)

- Настоящая аутентификация (пароли, сессии с истечением, роли через БД-права) —
  сейчас просто выбор пользователя без пароля.
- Telegram-уведомления.
- Отчёты и экспорт в Excel (`отчетность.md`) — модель данных это поддерживает,
  но экраны отчётов не сделаны.
- UI не адаптирован под мобильные экраны, стили минимальны.
- Расширенные поля агентов/компаний-покупателей (банковские реквизиты и т.п. —
  см. "Планы на будущее" в `агенты.md`) — в модели заложены только текущие простые поля.

## Что реально проверено (не просто написано, а прогнано вживую)

Тестировалось через реальные HTTP-запросы к работающему серверу на PostgreSQL 16
(не докер в этом случае — окружение, где писался код, не имеет доступа к Docker Hub,
поэтому `docker compose` собран по стандартным практикам, но не собирался здесь как
образ; сама логика приложения проверена полностью):

- Полный happy path: заявка на платёж через агента — создание → подача → назначение
  исполнителя → запрос уточнения → ответ → завершение исполнения → подтверждение
  Заказчиком → попытка закрыть без документов (отказано, показан список недостающих)
  → загрузка всех обязательных документов → успешное закрытие.
- Ветка отклонения: заявка на консультацию — подача → отклонение с причиной →
  попытка постороннего пользователя подтвердить ознакомление (отказано) → подтверждение
  автором заявки → переход в Архив.
- Делегирование: обычный Исполнитель без делегирования не может назначать исполнителя
  на чужую заявку (отказано); тот же пользователь с активной записью в `delegations`
  — может, и это явно помечается в истории статусов.
- Аудит-лог: после всех сценариев проверено содержимое таблицы `audit_log` — каждый
  переход статуса записан с корректным пользователем и значениями.

## Известные ограничения окружения при разработке

Приложение писалось и тестировалось в песочнице без доступа к Docker Hub, поэтому
`docker compose up --build` не был выполнен вживую в момент написания. Файлы
(`Dockerfile`, `docker-compose.yml`, `entrypoint.sh`) собраны по стандартным
паттернам и должны завестись без правок, но при первом запуске у тебя стоит
внимательно посмотреть вывод `docker compose up` на случай мелких накладок
(например, прав на volume с `uploads`).
